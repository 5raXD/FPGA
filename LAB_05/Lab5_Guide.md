# FPGA Lab 5 – Keyboard (PS/2 Interface) — Study & Build Guide

> **In one sentence:** In Lab 4 you built a **stopwatch**. In Lab 5 you build a
> **PS/2 keyboard scancode viewer** — press a key on a numeric keypad and the
> FPGA shows that key's 8‑bit scan‑code as 2 hex digits on the 7‑segment display
> and blinks an LED once per press.

This file collects (1) the **theory you must know** (it is also the test
material), (2) **the files you must write** and their ports, (3) the **order of
work**, and (4) **what to expect when you are done**.

---

## 0. The big picture (what the final thing does)

```
 [Numeric keypad] --PS2Clk/PS2Data--> [Ps2_Interface] --scancode[7:0]--> [Ps2_Display] --> 7-seg (2 hex digits)
                                            |                                            --> dp (dot for d/b)
                                            +----------- keyPressed ------------------>  --> led (1 short blink)
```

- Press `5` on the keypad → the two right 7‑seg digits show its scan‑code (`5` → `2E`).
- Every new key press → the LED gives one short visible blink (a *strobe*).
- The displayed code **stays** until the next key is pressed.
- Extended keys send a 4‑symbol code prefixed with `0xE0` → you must **ignore the
  `E0`** and show only the last 2 hex symbols.

This is the Lab‑5 equivalent of "the stopwatch counts and displays time" from Lab 4.

## 1. Theory you must know (this is the test material)

The PDF's "Lab Goals" lists exactly two topics. Know them well.

### 1.1 Asynchronous reset — advantages & disadvantages

A reset can be applied **synchronously** or **asynchronously**:

```verilog
// Synchronous reset  – reset only acts on a clock edge
always @(posedge clk)
    if (rst) q <= 0; else q <= d;

// Asynchronous reset – reset acts the instant it is asserted
always @(posedge clk or posedge rst)   // <-- rst in the sensitivity list
    if (rst) q <= 0; else q <= d;
```

| | Asynchronous reset | Synchronous reset |
|---|---|---|
| **Advantage** | Resets the design **immediately**, even when there is **no clock** (critical here — the PS/2 clock is idle most of the time). Does not need a running clock. | Easy static‑timing analysis; reset is "just another signal"; no special routing. |
| **Disadvantage** | **Reset *removal* (de‑assertion) is dangerous**: if reset is released too close to a clock edge it can violate recovery/removal time and cause **metastability**. Needs a *reset synchronizer*. Uses dedicated async reset routing. | Needs a **toggling clock** to take effect — useless while the clock is idle; reset pulse must be wider than one clock period to be captured. |

**Why it matters in this lab:** the keyboard clock (`PS2Clk`) only toggles during
a transmission and is idle the rest of the time. A *synchronous* reset on that
clock domain would not work while the clock is idle, so logic driven by `PS2Clk`
typically uses an **asynchronous, active‑low reset (`rstn`)** — note the figure
labels `rstn` and the `~reset` inversion of the push‑button.

See `LAB_05/Metastability.pptx` for the supporting theory on metastability /
recovery‑removal that motivates async‑reset synchronizers.

### 1.2 PS/2 interface — the protocol

The PS/2 keyboard is a **serial, device‑driven** interface using two wires:

- **`PS2Clk`** — the clock, **generated by the keyboard** (≈ 10–17 kHz, period
  ≈ 60–100 µs). **It is NOT a free‑running clock** — it toggles *only* while a
  byte is being transmitted, then stays idle (high).
- **`PS2Data`** — the serial data line, also driven by the keyboard.

**Frame structure (one byte = 11 bits, sent LSB first):**

| Bit | 0 | 1–8 | 9 | 10 |
|-----|---|-----|---|----|
| Meaning | **Start** (always `0`) | **Data D0…D7** (LSB first) | **Parity** (odd) | **Stop** (always `1`) |

- **Sampling rule:** the host (your FPGA) reads `PS2Data` **on the falling edge of
  `PS2Clk`** — the data is stable there.
- **Odd parity:** the parity bit makes the total number of `1`s among the 8 data
  bits + parity bit odd (used for error checking; you may verify it or ignore it).

**Scan‑codes (key events):**

- **Make code** — sent when a key is **pressed** (e.g. key `5` on the keypad → `0x2E`).
- **Typematic repeat** — while a key is **held**, the make code is **re‑transmitted
  repeatedly**. You must generate `keyPressed` **only once**, on the first press.
- **Break code** — sent when a key is **released**: `0xF0` followed by the make code.
- **Extended keys** — some keys send a **4‑symbol** code prefixed with `0xE0`
  (e.g. `0xE0 0x??`). You must **ignore the `0xE0` prefix** and display only the
  trailing make‑code byte.

**Identifying "a key was pressed" (`keyPressed`):** a single‑cycle pulse generated
the moment the **first make‑code frame** of a new press has been received — *not*
on each typematic repeat, and *not* on `0xE0`/`0xF0` packets.

### 1.3 The two key design subtleties

1. **Non‑toggling clock → decode early.** Because `PS2Clk` stops after the byte,
   you **must not wait for the whole frame's stop bit**. Decode the byte *before*
   the last cycle (note the hint: "evaluate the received bits every 11 clock
   cycles", "analyze the packet before receiving the stop bit"). If you wait for
   the stop bit, the clock is already idle and your logic can't fire.

2. **Two clock domains.** The receiver runs on the **slow `PS2Clk`**; the display
   runs on the **fast 100 MHz `clk`**. Crossing `scancode`/`keyPressed` from one
   to the other is a **Clock‑Domain‑Crossing (CDC)** — exactly why the PDF's Logic
   Analyzer tip says you cannot debug nets from different clock domains in one core.

### 2.1 `Ps2_Interface` — the receiver (the heart of the lab)

| Port | Dir | Width | Meaning |
|------|-----|-------|---------|
| `PS2Clk` | in | 1 | Keyboard clock (used **as a clock** in this module) |
| `rstn` | in | 1 | Active‑low **asynchronous** reset |
| `PS2Data` | in | 1 | Serial data line |
| `scancode` | out | 8 | Most recent make‑code byte |
| `keyPressed` | out | 1 | One pulse at the first make‑code of a new press |

**Strategy (the "easy" 22‑bit shift register method from the PDF):**

- A **22‑bit shift register** holds **two consecutive 11‑bit frames** (so you can
  see the *current* byte and the *previous* byte together — needed to spot `0xE0`
  and `0xF0` two‑frame sequences).
- On **each falling edge of `PS2Clk`**, shift `PS2Data` into the register (LSB
  first) and increment a small **bit counter**.
- A complete frame puts the data byte at a fixed slot, e.g.
  `current_byte = shiftreg[8:1]` (`[0]`=start, `[9]`=parity, `[10]`=stop), and the
  previous byte at `shiftreg[19:12]`.
- **Decode early** (every ~11 edges, before the stop bit) — don't wait for 22 bits.
- Drive `scancode <= current_byte` for a normal make code; **ignore** a byte equal
  to `0xE0`; treat `0xF0` (and the byte after it) as a **release**, not a press.
- Generate **`keyPressed`** as a single `PS2Clk`‑cycle pulse on the first make
  code only. A common trick: compare the just‑received frame with the previous
  frame — a typematic **repeat** sends two *identical* frames (suppress the pulse),
  while a genuinely new/first press differs (emit the pulse). Make sure `0xE0`/`0xF0`
  never produce a pulse.

> ⚠️ This single‑pulse / ignore‑`E0` / suppress‑repeats behaviour is the **main
> thing you must prove in simulation** (see §4). Get it right on paper first.

### 2.2 `Ps2_Display` — the visual feedback

| Port | Dir | Width | Meaning |
|------|-----|-------|---------|
| `clk` | in | 1 | 100 MHz system clock |
| `rstn` | in | 1 | Reset (active‑low) |
| `keyPressed` | in | 1 | Pulse from `Ps2_Interface` |
| `scancode` | in | 8 | Byte to display |
| `seg` | out | 7 | 7‑segment cathodes (`a..g`) |
| `an` | out | 4 | 4 digit anodes (active low) |
| `dp` | out | 1 | Decimal point (used to disambiguate `d`/`b`) |
| `led` | out | 1 | Strobe — one short visible blink per key press |

What it does:
- **7‑seg:** show the two hex symbols of `scancode` on the **two right‑hand
  digits** (built on your adapted `Seg_7_Display`). The two left digits can be
  blank. The shown value **latches** until the next key press.
- **`d` vs `0` and `b` vs `8`:** light the **`dp`** dot when showing `D`/`B`, or
  use lowercase `d`/`b` glyphs in the decode table.
- **`led` strobe:** stretch the 1‑cycle `keyPressed` into a pulse long enough for
  the eye (~tens of ms). A small counter that you load on `keyPressed` and that
  holds the LED on while it counts down does this (similar idea to the Lab‑4
  `Debouncer`/`Counter` pulse stretching in `LAB_04/to_submit/`).

## 4. Simulation — what your testbench must prove

You **only simulate `Ps2_Interface.v`**. In the report, screenshot the waveform
and annotate the key transitions with arrows. Your bench must demonstrate:

1. **Correct framing/decoding** — drive a full 11‑bit frame (start `0`, 8 data LSB
   first, odd parity, stop `1`) for a known byte and check `scancode` equals it.
2. **`keyPressed` fires once, at the first make code** — exactly at the moment the
   first make‑code frame completes.
3. **No extra pulses on typematic repeats** — send the same make code several times
   in a row; `keyPressed` must pulse **only once**.
4. **`0xE0` is ignored** — send an extended sequence `0xE0` then a byte; the `0xE0`
   must **not** drive `scancode` and must **not** pulse `keyPressed`; only the
   trailing byte shows.
5. **Reset behaviour** — assert `rstn=0` and confirm the registers clear (and that
   it works even with the PS/2 clock idle — the async‑reset advantage).

Helper you'll want in the bench: a task that, given an 8‑bit value, computes odd
parity and bit‑bangs the 11 bits onto `PS2Data` with a toggling `PS2Clk`.

```verilog
task send_byte(input [7:0] b);
    integer i;
    reg parity;
    begin
        parity = ~^b;            // odd parity
        ps2_bit(1'b0);           // start
        for (i = 0; i < 8; i = i+1) ps2_bit(b[i]);  // LSB first
        ps2_bit(parity);         // parity
        ps2_bit(1'b1);           // stop
    end
endtask
// ps2_bit() = put data on PS2Data, then pulse PS2Clk low->high with ~40us each.
```

## 6. What to expect when you're done (the "stopwatch moment")

- Plug the **numeric keypad** into the Basys3 **USB host** port, JTAG to the PC,
  program the FPGA.
- Press **`5`** → the two right 7‑seg digits show **`2E`**; press **`1`** → shows
  **`16`**; etc. (each numpad key has its own make code).
- Each press → the **LED blinks once** (short strobe), even if you hold the key
  (held key = repeats, but only the press is reported once).
- The displayed code **stays put** until the next key press.
- Extended keys (e.g. keypad `Enter`, `/`) display only their last 2 hex symbols —
  the `E0` is invisible.

That working "press a key → see its hex code + one blink" is the deliverable,
exactly like "press start → the stopwatch counts" was in Lab 4.

## 7. Common pitfalls (from the PDF's design notes)

- **Waiting for the full frame** → clock goes idle before you act. **Decode early**
  (before the stop bit / every ~11 edges).
- **Multiple `keyPressed` pulses** from typematic repeats → suppress repeats.
- **Forgetting to ignore `0xE0`** → wrong/extra display updates.
- **Using a synchronous reset on the `PS2Clk` domain** → reset never takes effect
  while the keyboard clock is idle. Use **async `rstn`**.
- **CDC / Logic Analyzer:** the slow `PS2Clk` nets and the fast `clk` nets are
  **different clock domains** — debug them in **separate** ILA cores (or two cores),
  never mixed in one.
- **Missing file headers** with both names → fails the submission rule.
