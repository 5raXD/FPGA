1. Vivado Installation

  1. Download Vivado 2023.2 ML Edition. The download link redirects to free Xilinx account creation.
  2. Run the installer. If a newer version is offered, click Continue, then Next.
  3. Enter Xilinx username and password. Choose “Download and Install Now”.
  4. Select Vivado ML Standard and click Next.
  5. On the component-selection screen, keep the same checkboxes as the lab's reference screenshot — most importantly, keep Artix-7 checked.
  6. Accept all license agreements.
  7. Confirm installation path, then proceed.
Popular installation bug: on some PCs, after installing Vivado, it won't run but keeps showing a request to update Visual Studio. Ignore these warnings, then go to the Vivado installation directory and remove the file named xvcredist.exe.

2. Vivado Project Creation

Launch Vivado, go to File → New Project, then walk the wizard:

  1. Project Name: e.g. lab1, choose a location.
    Lab rule: the project location must be on drive G if you're working in the lab right now, and the path must not contain spaces or Hebrew characters.
  2. Project Type: leave as RTL project, uncheck “Do not specify sources at this time”.
  3. Add Sources (next 3 screens):
    (1) Add design/sim sources(2) Add IP(3) Add constraints
    If you have sources ready, add them now. Check “copy sources into project” — otherwise they'll be left outside the project directory. Mostly skipped in the lab. Add the .xdc file from the lab team. Check “copy constraints files into project”.
    To add more sources later: Project Manager → Add Sources.
  4. Default Part: in the search box type the catalog name — xc7a35tlcpg236-2L . This is our chip. Select the (1 match) result, click Next, then Finish.

3. Constraints file (XDC)

The XDC file lets a designer present “special requests” to the otherwise-automatic synthesis/implementation flow. In our lab we use only two kinds:

Timing constraints

Define clock constraints, e.g. a 10 ns period (50% duty cycle):

create_clock -period 10 -name sys_clk_pin -waveform {0 5} -add [get_ports clk]

After timing analysis, see whether constraints are met in (Synthesis/Implementation) → Report Timing Analysis in the Flow Navigator.

IO constraints

Connect the FPGA package pins to the top-module ports. Example: tie pin W5 (oscillator crystal output) to port clk at LVCMOS33:

set_property PACKAGE_PIN W5 [get_ports clk]
set_property IOSTANDARD LVCMOS33 [get_ports clk]
Note: Ports that are not used in your design must be commented out.

4. Behavioral Simulation

Logical (behavioral) simulation tests correctness before the slow synthesis/implementation flow. Suppose you wrote a module FA.v and a test-bench FA_tb.v (with an initial block).

  1. Choose the test bench: Simulation Settings → Simulation top module → pick FA_tb → OK.
    Alternatively: right-click FA_tb.v under Simulation Sources and choose Set as Top.
  2. Run: Simulation → Run Simulation → Run Behavioral Simulation.
    Popular simulation problem: on some PCs, clicking “Run Behavioral Simulation” freezes Vivado at "Executing simulate step". Usually caused by antivirus — disable it or add Vivado to the exception list.
  3. After elaboration, a wave-diagram view opens:
    1. Click zoom-fit to set a decent time scale.
    2. Watch the TCL console for errors.
    3. Run more simulation steps if $finish wasn't reached in your test-bench.
    4. Re-run from start after modifying the test-bench or UUT.
    5. Drag more signals onto the waveform area to inspect them.

5. Implementing the design and checking timing

Timing analysis can run after synthesis (path delays over the synthesized netlist), but running it after implementation is more realistic — it includes FPGA placement and routing delays.

After Implementation → Run Implementation, always check errors and critical warnings, then the Timing tab at the bottom.

  • Setup paths: potential setup-time violations. Example: Path 1 with 4.646 ns net-delay + 1.414 ns logic-delay = ~6.06 ns total, with a positive slack of 3.936 ns — meaning you could hypothetically reduce the clock period by ~3.936 ns and still be correct.
  • Hold paths: the opposite — too-short paths. Hold violations occur when a path doesn't have enough delay to keep the contaminating signal alive through the hold window.

6. Integrated Logic Analyzer Setup

Vivado's ILA is a debug-core added alongside your design, letting you watch internal signals (FF outputs, wires, etc.) live from the actual FPGA on your screen.

After running synthesis: Synthesis → Set up debug. In the wizard's “Nets to debug” screen, choose nets either (1) from the netlist, or (2) by Ctrl-clicking nets in the synthesized schematic.

Notes:

  1. Because ILA is added after synthesis, optimization may have renamed nets.
  2. Don't choose the clock — it's recognized automatically.
  3. Don't pick nets before input-buffers or after output-buffers — they're non-debuggable.

Click the + button to add selected nets. On the “ILA Core Options” screen, choose the number of samples to be stored when the trigger occurs (e.g. 1024). The more signals and the longer the depth, the more on-chip memory is consumed — placement may fail if resources are exhausted.

Click Next, then Finish. Debugged nets appear as dashed lines.

7. Integrated Logic Analyzer Removal

Go to Synthesis → Set up debug, and on the first wizard screen choose “Disconnect all nets and remove debug cores.” Re-run synthesis.

If synthesis fails, manually remove the ILA-related lines from the XDC — start at create_debug_core u_ila_0 ila and remove everything below.

8. Integrated Logic Analyzer Usage

After successful ILA setup, implementation and device programming, Hardware Manager auto-detects the ILA core. Set up trigger signals (1, 2, 3), then (4) set the trigger to OR so any one trigger fires.

  1. Add the signals you want to monitor.
  2. Set triggers (e.g. react on exiting from 0 level) and arm the ILA core.
  3. The ILA now waits for a trigger.
  4. Hit a button on the board to fire the trigger — samples from the debugged nets appear on the wave-diagram view.

9. Programming the FPGA

Step 1 — Generate bitstream

After synthesis & implementation succeed: Program and Debug → Generate Bitstream. The Artix-7 bitstream is typically 17,536,096 bits long and can be compressed.

Step 2 — Connect Basys3 to PC

  • USB extender cable (on the desk, popping out from the monitor-cables hole).
  • Pay attention to JTAG cable orientation when inserting into the on-board port.
  • Power on the board with the power switch.
  • In Vivado: Program and Debug → Hardware Manager → Open Target → Auto Connect.

Step 3 — Programming mode (set by JP1 jumper)

ModeJP1 positionNotes
JTAG preferred upper Volatile. FPGA loses config on power-off. Generate .bit, then in Hardware Manager right-click the device → Program Device → select the .bit (usually pre-filled; otherwise look under lab1.runs/impl_1).
QSPI not used in lab middle Two-step. Set JP1 to JTAG, go to Project Properties → Bitstream, check -bin_file, regenerate the bitstream. Then Add Configuration Memory → select memory s25fl032p → agree to program → pick the .bin → OK. Non-volatile. When the DONE LED lights, power off, switch JP1 to QSPI (middle), and power on — FPGA auto-loads from flash in < 1 s.
USB not used in lab bottom Format a USB drive as FAT32, place a single .bit file in the root, attach to the J2 USB-HID port, set JP1 to USB (bottom), and push the PROG button or power-cycle the Basys3.

10. Warnings and Errors

Common warnings, critical warnings and errors in Vivado:

  1. [Synth 8-5788] Register X in module M has both Set and reset with same priority — your always@ block doesn't cleanly separate reset vs non-reset behaviour.
  2. [Synth 8-689] width (N) of port connection 'X' does not match port width (K) of module 'M' — wire X's bit-width doesn't match the port. Check buses/registers.
  3. [Synth 8-3848] Net 'X' in module/entity 'M' does not have driver — floating wire, or port 'X' wasn't connected.
  4. [Synth 8-3295] tying undriven pin M:X to constant 0 — synthesizer couldn't find X's functionality, ties it to 0.
  5. [Synth 8-3331] design M has unconnected port X — forgot to wire IO port X of module M.
  6. [Synth 8-3332] Sequential element (M/X_reg) is unused and will be removed from module M — declared but unused register, optimized out.
  7. [Synth 8-327] inferring latch for variable X — combinational logic with incomplete if/else or case without default.
  8. [Synth 8-5788] (same code, different cause) — X had no proper initialization value. Synchronous blocks should look like: 
    if (reset) begin
    // ALL FLIP_FLOPS INITIALIZATIONS
end else begin
    // ALL THE LOGIC
end
    Initialize all registers this way. The warning fires when X is assigned outside this if/else.
  9. [Vivado 12-584] No ports matched 'clk' — XDC refers to ports that don't exist in the top-level. Check the correct module is marked as top.
  10. … multi-driven net Q … — same register assigned from two separate always@ blocks.
  11. Found timing loop — combinational logic forms a cycle (not allowed). Vivado points at a random spot on the loop; use that hint to find where to insert at least one flip-flop. Often these loops cross several modules and are hard to fix.
