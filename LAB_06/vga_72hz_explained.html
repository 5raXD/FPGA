How do we know the design writes 72 frames per second?

VGA timing walk-through for VGA_Interface.v — FPGA Lab 6, BASYS3 (800×600 @ 72 Hz)

The one idea to remember: Vsync pulses exactly once per frame. So “do we write 72 frames/sec?” is the same question as “does Vsync fire 72 times per second?” The frame rate is just the Vsync frequency.

1. The clock chain

The frame rate falls out of one division: the pixel clock divided by the number of pixel-clocks in a frame. First, the pixel clock:

100 MHzboard oscillator (clk)
→
÷ 2pix_en toggles every cycle
→
50 MHzpixel clock (20 ns/pixel)
// 50 MHz pixel-clock ENABLE (a tick every other 100 MHz cycle)
reg pix_en = 0;
always @(posedge clk) pix_en <= ~pix_en;

Every counter and output advances only on else if (pix_en), so the design effectively ticks at 50 MHz — one pixel every 20 ns.

2. How many pixel-clocks are in one frame?

The counters wrap with `INC(x, max), which counts 0 … max-1:

  • Horizontal: h_count runs 0…1039 → 1040 pixel-clocks per line.
  • Vertical: v_count bumps once per line and runs 0…665 → 666 lines per frame.
1 frame = 1040 × 666 = 692,640 pixel-clocks

Note these are the totals (visible + blanking). The refresh rate depends only on the totals and the clock — not on where the sync pulses sit inside the line.

What a single line looks like

visible 800
FP 56
sync 120
BP 64
One horizontal line = 1040 pixel-clocks. (FP = front porch, BP = back porch.) The vertical frame has the same four-region structure: 600 + 37 + 6 + 23 = 666 lines.

3. The arithmetic

Tframe = 692,640 × 20 ns = 13,852,800 ns = 13.85 ms
frefresh = 50,000,000 Hz ÷ 692,640 = 72.19 Hz ≈ 72 fps

So you should see Vsync rising edges 13.85 ms apart.

5. How to actually verify it

A — In simulation (Icarus): time the Vsync edges

Instantiate the design and measure the gap between two Vsync rising edges. You should see period = 13.853 ms -> 72.19 Hz.

`timescale 1ns/10ps
module tb_framerate;
  reg clk = 0, rstn = 0;
  wire Hsync, Vsync;
  wire [3:0] r,g,b;
  wire [10:0] xc, yc;

  VGA_Interface dut(
    .clk(clk), .rstn(rstn), .pixel_color(12'h0),
    .vgaRed(r), .vgaGreen(g), .vgaBlue(b),
    .Hsync(Hsync), .Vsync(Vsync), .XCoord(xc), .YCoord(yc));

  always #5 clk = ~clk;        // 100 MHz -> 10 ns period
  initial #100 rstn = 1;

  real t_prev; integer n = 0;
  always @(posedge Vsync) begin
    if (n > 0) $display("frame %0d: period = %0.3f ms -> %0.2f Hz",
        n, ($realtime - t_prev)/1.0e6, 1.0e9/($realtime - t_prev));
    t_prev = $realtime; n = n + 1;
    if (n > 4) $finish;
  end
endmodule

B — Faster check: verify the two factors separately

  • Count Hsync pulses between two Vsync pulses → should be exactly 666.
  • Count pixel-clock ticks between two Hsync pulses → should be exactly 1040.

666 × 1040 × 20 ns = 13.85 ms ⇒ 72.2 Hz. Same answer, but each piece simulates quickly.

C — On real hardware

  • Put a scope or logic analyzer on the Vsync pin — it should read ~72 Hz.
  • Open the monitor's on-screen menu — it will report the detected mode, e.g. “800×600 @ 72 Hz.” If it locks and shows a stable image at all, your sync timing is within tolerance.
Bottom line: 72 fps holds because (50 MHz pixel clock) ÷ (1040 × 666 pixel-clocks per frame) = 72.2 Hz, and those numbers are pinned by the pix_en divider and the H_TOTAL / V_TOTAL constants. Counting Vsync pulses — in simulation or with a scope — confirms it directly.
