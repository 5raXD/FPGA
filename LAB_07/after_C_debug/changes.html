LAB_07 — Snake: after_C_debug

What was broken in the original sources, why, and exactly what I changed. All edits live only in after_C_debug/ — the original files in LAB_07/ were left untouched.

Verification. The fixed set was compiled and elaborated with Vivado's xvlog + xelab (2023.2): 
xvlog *.v          → all modules analyzed, 0 errors
xelab snake_game   → "Built simulation snapshot work.snake_game"  (EXIT 0)
The 6 errors / 1 critical warning from the original synthesis run are gone.
important "Generate Bitstream takes forever / Vivado thrashes"

The first round of fixes synthesized correctly but synthesis thrashed (RAM full, swap 100%, repeated Thrashing Detected! in the log) and never reached implementation. Root cause: the original snake stored a 100×75 grid of 13-bit countdown cells — ~97,500 flip-flops plus a subtractor on every one of the 7,500 cells each tick. That is far too much logic for the xc7a35t and blew up memory during optimization.

Fix (v2): snake.v was rewritten to store the snake the way a working reference design does — as a short list of segment coordinates moved like a shift register. See §5. Same module interface, so nothing else in the project changed. Resource use drops from ~97.5k FFs to ~1.4k.

1 · The error that actually failed synthesis

Synth 8-524 / 8-6156 snake.v:117

Part-select [15:0] out of range of prefix 'length'.

length is declared reg [$clog2(GRID_X*GRID_Y)-1:0]. With 100×75 = 7500, $clog2(7500) = 13, so length is only 13 bits ([12:0]). Slicing length[15:0] reaches bits 13–15 that don't exist → Vivado errors and refuses to synthesize Snake (that's the 8-6156 failed synthesizing module 'Snake' roll-up).

- assign score = length[15:0];
+ assign score = length;        // 13-bit length zero-extends into 16-bit score

The output port is [15:0] score; assigning a 13-bit value zero-extends it automatically — no slice needed.

2 · Pixel_Painter port mismatch

Synth 8-7023 Synth 8-7071 snake_game.v:102 / Pixel_Painter.v

"14 connections declared, but only 9 given" and 5 × "port unconnected".

Pixel_Painter declared 14 ports but the top instance wired only 9 — the snake/food signals (crash, is_food, on_snake, is_head) were commented out, plus an unused start_game. With those inputs floating, GridMapper would never see the snake or food, so nothing would be drawn even after synthesis passed.

Fix:

  • Pixel_Painter.v — removed start_game (it is genuinely unused inside the module).
  • snake_game.v — declared a wire is_food; and connected every game signal:
Pixel_Painter #(...) painter(
    .clk(clk), .reset(reset), .tick(tick), .keyPressed(keyPressed),
    .crash(crash), .is_food(is_food), .on_snake(on_snake), .is_head(is_head),
    .XCoord(XCoord), .YCoord(YCoord),
    .x(x), .y(y), .pixel_color(pixel_color));

3 · Missing welcome.mem

Synth 8-4445 grid_mapper.v:57

"could not open $readmem data file 'welcome.mem'". The file doesn't exist in the project, and the welcome array it filled was never actually used (the IDLE branch of the colour case was empty). Two problems in one: a missing file and an inferred latch (see §4).

- reg [GRID_X-1:0] welcome [0:GRID_Y-1];
- initial $readmemb("welcome.mem", welcome);

Removed the dead welcome array entirely and gave IDLE a real solid colour instead, so no .mem file is needed for the welcome screen.

Only skull.mem remains (for the game-over screen) — remember to add it to the Vivado project (Add Sources → Add or create design sources → add files of type Memory File) so synthesis can find it on its search path.

4 · GridMapper FSM & colour logic

logic buggrid_mapper.v

(a) The nested-assignment trap. The original transitions read:

IDLE: state <= keyPressed? PLAY : state <= IDLE;

The ?: needs an expression as its else operand, so Verilog parses the trailing state <= IDLE as the relational comparison "state ≤ IDLE", which yields 1'b1/1'b0. So in IDLE with no key pressed it wrote state <= (state≤0) = 1 = PLAY — it left IDLE on its own. It compiles cleanly because it's legal; it just doesn't mean what it looks like. Fixed all three branches to assign plain state values:

IDLE:      state <= keyPressed ? PLAY      : IDLE;
PLAY:      state <= crash      ? GAME_OVER : PLAY;
GAME_OVER: state <= keyPressed ? IDLE      : GAME_OVER;

(b) Reset went to the wrong state. state <= GAME_OVER; // fix me → state <= IDLE;

(c) Head was never drawn (priority bug). The head cell is also on the snake, so {on_snake,is_food,is_head} = 101 for the head — which fell through the case to the background colour. Replaced the flat case with a priority chain, and gave the head its own colour:

if      (is_head)  block_color = SNAKE_HEAD_COLOR; // yellow, now unique
else if (on_snake) block_color = SNAKE_COLOR;      // body (grey)
else if (is_food)  block_color = FOOD_COLOR;       // red
else               block_color = GREEN;            // background

(d) Inferred latch in IDLE. The IDLE case assigned nothing to block_color (a combinational reg), inferring a latch. IDLE now assigns a solid welcome colour.

5 · Snake rewritten as a segment list (the thrashing fix)

rewritesnake.v

Old representation (made synthesis thrash). The snake was a full grid reg [12:0] grid [0:74][0:99] where each cell held a countdown of how many ticks until it clears, and every cell was decremented each tick. That is ~97.5k FFs plus ~7,500 subtractors — far too much for the part, so optimization ran the machine out of memory.

New representation. Store only the body, as a list of segment coordinates (segment 0 = head), exactly like a known-good reference design:

reg [XW-1:0] sx [0:MAX_LEN-1];   // segment x coords   (MAX_LEN = 100)
reg [YW-1:0] sy [0:MAX_LEN-1];   // segment y coords
reg [LW-1:0] length;             // how many segments are visible

~1.4k FFs total. Movement is a shift register — every segment follows the one ahead of it — with length deciding how much of the list is drawn:

// shift the whole list, put the new head at [0]
for (i = MAX_LEN-1; i > 0; i = i-1) begin
    sx[i] <= sx[i-1];  sy[i] <= sy[i-1];
end
sx[0] <= next_x;  sy[0] <= next_y;
if (is_eaten && length < MAX_LEN) length <= length + 1;  // grow

Because the list is always shifted, the segment just past the tail already holds the old tail position — so growing is simply "increment length", no special-case needed.

Queries for the painter become cheap comparators (not a RAM):

on_snake : OR over i<length of (sx[i]==x && sy[i]==y)
is_head  : (sx[0]==x && sy[0]==y)
is_food  : (plant_x==x && plant_y==y)
score    : length - 1

Collision uses the combinational next-head position (so it is not a tick late) against the walls and the active body segments:

wire [XW-1:0] next_x = (dir==LEFT)? sx[0]-1 : (dir==RIGHT)? sx[0]+1 : sx[0];
wire [YW-1:0] next_y = (dir==UP)?   sy[0]-1 : (dir==DOWN)?  sy[0]+1 : sy[0];
... if (next_x>=GRID_X || next_y>=GRID_Y || hit_body) crash <= 1;

Unsigned wrap also catches the left/top walls (stepping off 0 wraps to a value ≥ GRID). All segments are reset to the centre, and is_food is now actually driven (it had no assign in the original). The module's ports are unchanged, so snake_game.v, Pixel_Painter.v and grid_mapper.v did not need to change.

vs. the reference design: same coordinate-list idea, but kept inside your modular structure — it uses the 7 Hz tick enable instead of a gated slow_clk, and keeps the Snake/Pixel_Painter/GridMapper split rather than drawing pixels inside the snake module.

6 · Navigation_System never turned

logic bugNavigation_System.v

The original wrote a scratch register dir_next on each key, but never copied it back to dir — dir was only assigned in reset. The snake was stuck going RIGHT forever. I dropped dir_next and update dir directly, keeping the anti-reversal guard:

8'h6B: if (dir != RIGHT) dir <= LEFT;   // numpad 4 = Left
8'h74: if (dir != LEFT)  dir <= RIGHT;  // numpad 6 = Right
8'h75: if (dir != DOWN)  dir <= UP;     // numpad 8 = Up
8'h73: if (dir != UP)    dir <= DOWN;   // numpad 5 = Down

Scancodes match the assignment's 4/6/8/5 = L/R/U/D mapping (numpad set).

7 · Welcome screen, Enter-to-start & restart

feature game_FSM.v, snake_game.v, grid_mapper.v, welcome.mem

Goal: when the board is programmed it should open on a welcome screen; press Enter to start; crash shows the skull; and the reset button should send you back to a fresh welcome screen.

(a) The FSM moved to the top level — new game_FSM.v. Previously the state machine lived inside GridMapper, so nothing else could see the game state. Now game_state is a top-level signal:

IDLE  --enter-->  PLAY  --crash-->  GAME_OVER  --enter-->  IDLE
(any state) --reset--> IDLE

(b) "Enter to start". In snake_game.v the Enter key (PS/2 scancode 0x5A) makes a one-cycle pulse that drives the FSM:

wire enter = keyPressed && (scancode == 8'h5A);

(c) Restart actually works now. The key trick: the Snake is held in reset whenever we are not playing, so every game begins fresh (length 1, centred, score 0, new food). Before, going back to PLAY after a crash kept the dead snake and instantly re-lost.

wire snake_rst = reset || (game_state != PLAY);
... Snake snake( .reset(snake_rst), ... );

So: btnC (reset) → welcome screen + fresh snake at any time; from the skull screen, Enter → welcome, Enter → new game.

(d) The welcome bitmap. GridMapper now shows a bitmap in IDLE, the same way it shows skull.mem in GAME_OVER:

reg [GRID_X-1:0] welcome [0:GRID_Y-1];
initial $readmemb("welcome.mem", welcome);
wire on_welcome = welcome[y][GRID_X-1-x];
... IDLE: block_color = on_welcome ? WELCOME_FG : WELCOME_BG;

welcome.mem is generated by welcome.py (a tiny 5×7 font rendered to a 100×75 bitmap) and reads "SNAKE / PRESS ENTER". Regenerate or edit the text with python3 welcome.py; preview is welcome.png.

Add to the Vivado project: both welcome.mem and skull.mem must be added as design sources (Memory Files) so synthesis finds them.

Known limitations / not yet done

  • Timing not met. The implementation reports negative slack (failing endpoints) — the per-pixel "is this cell on the snake?" check is a wide comparator chain over all segments feeding the pixel path. The game still works on hardware, but to close timing cleanly you'd register the cell queries (pipeline on_snake/is_head a stage) or shrink MAX_LEN.
  • Head-follows-tail edge. A cell that is about to clear this tick still reads non-zero during the collision check, so turning straight into the vacating tail counts as a crash. A classic snake nuance; harmless for normal play.
  • LFSR food distribution is non-uniform (see screen.png) and food can spawn on the snake — the food_on_snake avoidance path is stubbed out in the original and not addressed here.
  • Max length. The body list holds up to MAX_LEN = 100 segments; eating beyond that stops growing the snake (it caps). Raise the parameter if you want a longer maximum — cost grows linearly, not as the whole grid.
  • Renderer.v and GSM.v remain empty stubs and are not part of the build.
