What to do

The timing engine is done and verified in Lab 6. Copy it as-is; you do not touch its internals for Snake: 

// Copy this file into the lab7 sources:
LAB_06/VGA_Interface.v        // sync generator + scan coordinates
The full theory (raster scan, porches, the 1040×666 timing canvas, sync polarity) lives in the Lab-6 visual guide: → open the VGA Protocol Guide

Ports (already defined)

module VGA_Interface(
    input  wire        clk,            // 100 MHz
    input  wire        rstn,           // active-low reset
    input  wire [11:0] pixel_color,    // color for the pixel being scanned
    output reg  [3:0]  vgaRed,
    output reg  [3:0]  vgaGreen,
    output reg  [3:0]  vgaBlue,
    output reg         Hsync,
    output reg         Vsync,
    output wire [10:0] XCoord,         // current pixel column (0..1039)
    output wire [10:0] YCoord          // current pixel row    (0..665)
    );
SignalDirectionConnect in Snake to…
pixel_color[11:0]inRenderer output
XCoord, YCoordoutRenderer input (which pixel to color)
vgaRed/Green/Blue, Hsync, VsyncoutVGA pins (from the XDC)

How it plugs into the game

It forms a tiny feedback loop with the Renderer — identical to the Lab-6 Drawer↔VGA_Interface wiring (see LAB_06/VGA_Top.v for the exact pattern): 

VGA_Interface u_vga(
    .clk(clk), .rstn(rstn),
    .pixel_color(pixel_color),     // <- from Renderer
    .XCoord(XCoord), .YCoord(YCoord),  // -> to Renderer
    .vgaRed(vgaRed), .vgaGreen(vgaGreen), .vgaBlue(vgaBlue),
    .Hsync(Hsync), .Vsync(Vsync));
Don't defeat the blanking gate. This module already forces {R,G,B}=0 outside the visible 800×600 box. The Renderer just supplies a candidate color; never bypass the interface and drive the VGA pins yourself, or you get the rolling/tearing screen from Lab 6.
