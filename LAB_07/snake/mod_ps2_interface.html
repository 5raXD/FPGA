Full background on how it works (the 11-bit frame, falling-edge sampling, the one-shot pulse) is in the Lab-5 visual guide: → open the PS/2 Protocol Guide

Ports (already defined)

module Ps2_Interface(
    input  wire       PS2Clk,      // keyboard clock  (async, slow)
    input  wire       rstn,        // active-low reset
    input  wire       PS2Data,     // keyboard data   (async)
    output reg  [7:0] scancode,    // the received byte
    output reg        keyPressed   // one-shot pulse: "a new key arrived"
    );
SignalDirectionConnect in Snake to…
PS2Clk, PS2Datainthe two FPGA keyboard pins (from the XDC)
rstninyour active-low reset
scancode[7:0]outDirection Control
keyPressedoutDirection Control (use as an enable)

How it plugs into the game

You only consume two outputs. Every time keyPressed pulses high for one cycle, the byte on scancode is valid — hand both to Direction Control, which turns the numeric-keypad keys (4 6 8 5) into a movement direction.

Keep the one-shot. A held key auto-repeats its make-code every ~100 ms. keyPressed is already a single-cycle pulse, so don't re-derive an edge from scancode — just act on the pulse. (Lab-5 pitfall.)
Clock-domain note. PS2Clk/PS2Data are asynchronous to your 100 MHz logic. The interface samples them safely; if you ever resample its outputs in a different domain, copy the 3-FF synchroniser pattern from Ps2_Display.v (the Lab-5 metastability topic).
