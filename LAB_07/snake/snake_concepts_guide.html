1. What kind of lab is this? (No new protocol — integration)

Labs 5 and 6 each taught you one new protocol: PS/2 (how to read a keyboard) and VGA (how to paint a screen). Lab 7 teaches you no new protocol at all. Instead it asks the harder question every real digital-design course eventually asks:

The whole lab in one sentence: you already own all the pieces — a keyboard reader, a screen painter, a 7-segment display, and counters — now design the logic that turns those pieces into a game, from scratch, with no module skeletons handed to you.

That is exactly what the assignment says: "you are expected to design and implement the modules by yourselves without detailed guidance. We will not supply any code or module inputs and outputs." So this guide is not a protocol reference — it is a concept and architecture guide. Its job is to give you the mental models you don't have yet so you can invent the block diagram yourself.

What you are building

  • Graphics on a VGA monitor — the snake and the food, clearly distinguishable. (reuse Lab 6)
  • Score on the 7-segment display, incremented every time the snake eats. (reuse earlier 7-seg labs)
  • Control from the keyboard: 4=Left, 6=Right, 8=Up, 5=Down, with no immediate 180° reversal. (reuse Lab 5)
  • Game logic: the snake grows when it eats, and the game ends if it hits a wall or itself.
  • Bonus: a dedicated "Game Over" screen.
Mindset shift. In Labs 5–6 the timing was dictated by an external standard you had to obey. Here you invent the timing, the data structures, and the rules. The grade is mostly about a clean block diagram and modules that are parametric (assignment rule #3: e.g. grid size, snake max length, and bit-widths must be parameters, not magic constants).

3. From Pixels to a Game Grid

The screen is 800 × 600 = 480 000 pixels. You absolutely do not want to reason about the snake at pixel resolution. The single most important simplification in the whole lab is to quantize the screen into a coarse grid of cells, exactly like the downsample trick from Lab 6 (hint #8).

Pick a cell size that divides nicely. With 20 × 20-pixel cells you get a 40 × 30 grid (40·20 = 800, 30·20 = 600). The snake then lives on a tiny 40×30 board — small enough to store, fast enough to render.

The math behind a "clean" cell size: gcd(800, 600)

A square cell of side d tiles the visible area with no leftover pixels only if d divides both 800 and 600 — that is, only if d divides their greatest common divisor: 

gcd(800, 600) = 200          // 800 = 200·4 , 600 = 200·3

So 200 × 200 is the largest square cell that fits the screen exactly (a 4 × 3 board — far too coarse to play). Every smaller clean choice is a divisor of 200; pick one small enough to be playable:

Cell d (divisor of 200)Grid (cols × rows)Total cellsNote
2004 × 312largest possible — too coarse
1008 × 648still very chunky
5016 × 12192tiny, trivial to simulate
4020 × 15300good for a first bring-up
2532 × 24768nice playable size
2040 × 301200the sweet spot used in this guide
1080 × 604800fine — bigger body memory
8100 × 757500fine
5160 × 12019200very fine — lots of cells

(The full divisor list is 1, 2, 4, 5, 8, 10, 20, 25, 40, 50, 100, 200 — each gives a grid of 800/d × 600/d.)

Smarter sizing: use log₂ so "divide" becomes a shift

Converting a pixel to a cell is a division: col = XCoord / d. Division is expensive in hardware — unless d is a power of two. Dividing by 2^k is simply dropping the low k bits, i.e. a wire-only shift, where k = log₂(d): 

// if d = 2^k  then  X / d  ==  X >> k ,  with  k = log2(d)
//   d=8  -> k=3 -> XCoord >> 3   (no divider, just re-wiring!)
//   d=16 -> k=4 -> XCoord >> 4
//   d=32 -> k=5 -> XCoord >> 5

For a pure shift, log₂(d) must be a whole number — i.e. d ∈ {2, 4, 8, 16, 32, …}.

The smartest cell size satisfies both rules at once:

  • divisor of 200 → tiles the screen with no leftover pixels (the gcd rule), and
  • power of two → the cell index is a free shift (the log₂ rule).

Factor the gcd: 200 = 2³ × 5² = 8 × 25. The only powers of two hiding inside it are 1, 2, 4, 8 — so the largest size meeting both wishes is 8.

Best of both worlds: d = 8. It tiles exactly (800/8 = 100, 600/8 = 75 → a 100 × 75 grid) and its index is a pure shift (col = XCoord >> 3, row = YCoord >> 3) because log₂8 = 3. No divider, no leftover pixels.
Cell dlog₂ dIndex opDivides 200? (clean tile)Grid
83>>3✓ yes100 × 75
164>>4✗ no (600/16 = 37.5)50 × 37 + blank strip
20≈ 4.32real divide✓ yes40 × 30
25≈ 4.64real divide✓ yes32 × 24
325>>5✗ no25 × 18 + blank strip
The trade-off in one line. A divisor of 200 (20, 25, 40…) tiles perfectly but needs a real divider; a power of two (16, 32) costs only a shift but leaves a partial row/column to blank. d = 8 is the rare size that is both — and is why the 100 × 75 grid is the cheapest clean choice. (Powers of two > 8 keep the shift but waste an edge strip; this is the caveat behind the 16-px option below.)

The two coordinate systems

SpaceRangeUsed by
Pixel spaceX 0–799, Y 0–599 (from VGA XCoord/YCoord)the Renderer, per pixel-clock
Cell spacecol 0–39, row 0–29the Snake_Logic brain, per tick

Converting between them is just a shift (because 20 isn't a power of two, you can either divide, or — easier in hardware — pick a power-of-two cell size like 16 and use a shift):

// Option A: 20px cells -> divide (synthesizes to a small divider/compare logic)
wire [5:0] cell_col = XCoord / CELL;   // CELL = 20  -> 0..39
wire [5:0] cell_row = YCoord / CELL;   //              0..29

// Option B (preferred): 16px cells -> just shift, no divider
//   800/16 = 50 cols, 600/16 = 37.5 -> use 37 rows (592px) and blank the rest
wire [5:0] cell_col = XCoord >> 4;     // 0..49
wire [5:0] cell_row = YCoord >> 4;     // 0..36
Make the grid parametric (assignment rule #3). Define CELL, GRID_W, GRID_H as parameters. Then a grader can shrink the grid to make simulation tiny, and your collision/render logic still works untouched.

4. Representing the Snake in Hardware

This is the conceptual heart of the lab and the part with no obvious "right" answer. The snake is a growing, ordered list of occupied cells with a head and a tail. Software would use a queue or linked list — but you have no malloc in hardware. Here are the two standard approaches; pick one and justify it in your report.

Approach A — Array of segment coordinates (a shift register of cells)

Store an array of up to MAX_LEN coordinates. Index 0 is the head. On every tick, every segment copies the position of the segment ahead of it, and the head steps in the current direction. 

localparam MAX_LEN = 64;
reg [5:0] snake_x [0:MAX_LEN-1];   // column of each segment
reg [5:0] snake_y [0:MAX_LEN-1];   // row    of each segment
reg [6:0] length;                  // how many segments are alive

integer i;
always @(posedge clk) if (tick) begin
    for (i = MAX_LEN-1; i > 0; i = i-1) begin   // body follows the head
        snake_x[i] <= snake_x[i-1];
        snake_y[i] <= snake_y[i-1];
    end
    snake_x[0] <= snake_x[0] + dx;              // head steps in current dir
    snake_y[0] <= snake_y[0] + dy;
    if (ate_food) length <= length + 1;         // growth = keep one more segment
end
  • + Simple to think about — it literally is the snake.
  • − Rendering cost — to color a pixel you must compare its cell against all live segments (up to MAX_LEN comparators), and self-collision is the same all-pairs check.
  • Best when MAX_LEN is modest (say ≤ 64).

Approach B — Grid memory with a countdown (the scalable trick)

Keep a 2-D memory, one entry per cell, holding a small number: 0 = empty, and any non-zero value n means "this cell is body, and it will become empty in n more ticks." The genius is that the tail clears itself automatically.

On each tick, write the current snake length into the new head cell.
Decrement every non-zero cell by 1. The oldest cell (the tail) was holding 1, so it hits 0 and vanishes.
A cell is "snake" simply if its value is non-zero — that's your render test and your self-collision test, both O(1) per cell.
Ate food? Don't shrink: bump length by 1 so every cell survives one extra tick.
  • + O(1) render & collision — just read one cell.
  • − "decrement every cell" each tick — but ticks are ~100 ms apart and the FPGA runs at 100 MHz, so you have millions of clock cycles to walk the grid one cell at a time. Plenty of time.
  • Scales to large snakes and big grids; uses block RAM cleanly.
Recommendation. For a first working version, Approach A with a small MAX_LEN is easiest to get on the screen and to simulate. If you have time and want a long snake, Approach B is the elegant, scalable answer — and makes a great point in the "internal design and logic" part of your report.

5. Game Time: the "Tick" vs. the Pixel Clock

There are three different rates living in this design, and confusing them is the classic beginner mistake:

Clock / rateFrequencyDrives
System clock100 MHzeverything synchronous
Pixel clock50 MHz (÷2)VGA scanning, the Renderer
Game tick~5–10 Hzthe snake moves one cell

The snake must move at a speed a human can play — a handful of cells per second. If you moved it on every pixel clock it would cross the screen in microseconds. So you build a clock-enable pulse: a slow strobe that is high for exactly one 100 MHz cycle every ~150 ms, and your whole game logic advances only when that strobe is high. 

// Game tick: one 100 MHz cycle high every TICK_MAX cycles.
// 100e6 * 0.15 s = 15_000_000 -> ~6.7 moves/second.
localparam TICK_MAX = 15_000_000;
reg [23:0] tick_cnt = 0;
reg        tick = 0;
always @(posedge clk) begin
    if (tick_cnt == TICK_MAX-1) begin tick_cnt <= 0; tick <= 1; end
    else                        begin tick_cnt <= tick_cnt + 1; tick <= 0; end
end
Why a clock-enable and not a real slow clock? Generating a genuine 6 Hz clock and using it as a posedge creates a new, unsynchronized clock domain (and burns a clock buffer). The clean FPGA idiom is to stay in one 100 MHz domain and gate updates with if (tick). Same reasoning the VGA lab used for the 50 MHz pixel clock-enable.
Optional polish: make the snake speed up as the score grows by shrinking TICK_MAX — a nice, cheap difficulty curve and another use of the "make it parametric" rule.

6. Control: Keyboard, Direction & No Reversal

Reuse your Lab-5 PS2_Interface. It already hands you a scancode and a one-shot keyPressed pulse. You only need to map four numeric-keypad keys to four directions. The make-codes are exactly the ones from the Lab-5 keypad table:

KeyDirectionKeypad make-code(dx, dy)
4Left0x6B(−1, 0)
6Right0x74(+1, 0)
8Up0x75(0, −1)
5Down0x73(0, +1)
Mind the screen's Y axis. On a VGA raster, row 0 is the top and the row index grows downward. So "Up" means decreasing the row (dy = −1) and "Down" means increasing it (dy = +1). Getting this backwards is a common and confusing bug.

The "no immediate reversal" rule (requirement 3.b)

If the snake is moving Right, pressing Left must be ignored — otherwise the head walks straight into its own neck and the game ends instantly. The rule: reject any new direction that is the exact opposite of the current direction. 

localparam L=2'b00, R=2'b01, U=2'b10, D=2'b11;
reg [1:0] dir;       // current direction (changes only on a tick)
reg [1:0] dir_next;  // latest legal request from the keyboard

always @(posedge clk) if (keyPressed) begin
    case (scancode)
        8'h6B: if (dir != R) dir_next <= L;   // pressing Left is legal unless going Right
        8'h74: if (dir != L) dir_next <= R;
        8'h75: if (dir != D) dir_next <= U;
        8'h73: if (dir != U) dir_next <= D;
    endcase
end

always @(posedge clk) if (tick) dir <= dir_next;  // commit the turn once per move
Latch the request, apply it on the tick. Notice dir only updates inside if (tick). If you turned the snake the instant a key was pressed, two fast key presses between ticks could let it reverse in two 90° steps within a single move. Buffering into dir_next and committing once per tick prevents that and keeps motion grid-aligned.

7. Food & Pseudo-Random Numbers (LFSR)

When the snake eats, you must drop new food at a "random" empty cell. An FPGA has no rand(), but you don't need true randomness — you need cheap, unpredictable-looking numbers. The standard hardware answer is a Linear-Feedback Shift Register (LFSR).

What the name means

  • Shift Register — a row of flip-flops; on every clock each bit shifts one position into its neighbour.
  • Feedback — the bit that would normally be lost off the end is replaced by a value fed back from inside the register.
  • Linear — that fed-back value is computed only with XOR gates (XOR is a linear operation over GF(2)). No carry, no addition — just a handful of XORs.

So an LFSR is just "shift everything by one, and fill the vacated bit with the XOR of a few chosen bits (the taps)." That one rule, iterated, marches the register through a long, scrambled-looking sequence of values that is actually 100% deterministic — perfect for a game, where you want variety but also repeatable simulation.

Why those exact taps? (maximal length)

You cannot pick the tap bits at random. Only certain tap sets make the register cycle through every non-zero value before repeating — the longest possible run, called a maximal-length sequence. An n-bit maximal LFSR has period 2n − 1; for n = 16 that is 65 535 distinct values. The tap set {15, 13, 12, 10} used here is one of the known-good "primitive polynomial" choices for 16 bits (you look these up in a tap table such as Xilinx XAPP052; you don't derive them by hand).

Never let the LFSR be all-zeros. If every bit is 0, the XOR of the taps is also 0, so it shifts in 0 forever — the register is stuck. That's why the period is 2n−1 (not 2n): the all-zeros state is excluded. Always initialise it to a non-zero seed (and if you ever clear it on reset, clear it to a non-zero constant). 
// 16-bit Fibonacci LFSR (taps 15,13,12,10), free-running on the system clock.
reg [15:0] lfsr = 16'hACE1;          // any NON-ZERO seed
always @(posedge clk)
    // shift left; the new bit 0 = XOR of the four tapped bits
    lfsr <= {lfsr[14:0], lfsr[15]^lfsr[13]^lfsr[12]^lfsr[10]};

// When food must respawn, sample the LFSR and squeeze it into the grid:
wire [5:0] food_col = lfsr[5:0]  % GRID_W;   // 0..GRID_W-1
wire [5:0] food_row = lfsr[11:6] % GRID_H;   // 0..GRID_H-1
Fibonacci vs. Galois form. The code above is the Fibonacci style: all taps are XORed together and fed into one end (it matches the diagram one-to-one, so it's easiest to learn from). There is an equivalent Galois style that instead XORs the shifted-out bit into several positions inside the register; it produces the same kind of sequence but with shorter logic paths. For this lab the Fibonacci form is perfectly fine — pick whichever you find clearer.
Food must not land on the snake. After generating a candidate cell, check whether it is already occupied. If it is, just keep sampling on the next clock (the LFSR is free-running, so the next value is different) until you find an empty cell, then latch it. Because the LFSR runs at 100 MHz, a valid spot is found in nanoseconds.
Why let the LFSR free-run? If you only clocked it when food is eaten, the "random" position would depend solely on how many foods you've eaten — perfectly predictable. Free-running means the value at the moment of eating depends on real time, which feels random to the player.

8. Collision Detection (Wall & Self)

The game ends on two events. Both are tested on the new head position, computed on each tick before you commit the move.

Wall collision

The head leaves the grid. With cell coordinates this is a trivial bounds check:

wire hit_wall =
     (new_col < 0) || (new_col >= GRID_W) ||
     (new_row < 0) || (new_row >= GRID_H);

(If your columns/rows are unsigned, "< 0" becomes a wrap to a huge value, which the >= check already catches.)

Self collision

The head lands on a cell that is already part of the body.

  • Approach A: compare (new_col,new_row) against every body segment.
  • Approach B: just read the grid cell — non-zero means "already snake," so it's a hit. (one memory read!)
One self-collision subtlety. The tail cell is about to be vacated this very tick, so moving the head into the old tail position is actually legal (the tail will have moved away). Decide whether your check happens before or after the tail update and document it — graders love this kind of detail in the report.

9. Rendering on VGA

Reuse the Lab-6 VGA_Interface unchanged — it still produces XCoord/YCoord and the sync signals. Your old "Drawer" becomes a Renderer whose only job, for the pixel currently being scanned, is to answer: "is this pixel snake, food, border, or background?" 

// Combinational color decision for the pixel at (XCoord,YCoord)
wire [5:0] col = XCoord / CELL;
wire [5:0] row = YCoord / CELL;

always @(*) begin
    if (game_over)              pixel_color = OVER_COLOR;   // game-over screen
    else if (col==food_col && row==food_row)
                                pixel_color = 12'hF00;      // food = red
    else if (is_snake(col,row)) pixel_color = 12'h0F0;      // snake = green
    else                        pixel_color = 12'h000;      // background = black
end

is_snake(col,row) is where the two representations differ: in Approach A it ORs together comparisons against each segment; in Approach B it is one memory read (grid[row][col] != 0). To make the head stand out (requirement 1.b — "clearly distinguishable"), color the head a lighter green.

head (light green) body (green) food (red) background (black)
Still obey the Lab-6 blanking rule. The VGA_Interface must force {R,G,B}=0 whenever hcount≥800 or vcount≥600. The Renderer only produces a candidate color; the interface gates it. Don't drop that gate or you get the rolling/tearing screen from Lab 6.
Reading vs. writing the grid. The Renderer reads the snake state at 50 MHz (per pixel); the Snake_Logic writes it at the tick rate. Because the writes happen in a one-cycle burst once every ~150 ms, there's no real contention — but if you use block RAM, give it a true read port for the Renderer and a write port for the logic (a simple dual-port pattern).

10. Score on the 7-Segment Display

This is pure reuse of your earlier 7-segment lab. Keep a score register, increment it by one each time ate_food fires, and feed it to your existing display module (the one that multiplexes the four digits via an[3:0] and drives seg[6:0]). 

reg [13:0] score;                     // up to 9999
always @(posedge clk) begin
    if (start)          score <= 0;
    else if (ate_food)  score <= score + 1;   // one point per food
end
// score -> (existing) binary-to-BCD / hex -> seg7_display module
Decimal vs. hex. If you display the raw binary in hex it's trivial; if requirement readers expect a human decimal score, add a binary-to-BCD step (double-dabble) before the digit mux. Either is acceptable — state which you chose.

11. Game State Machine & Game-Over Screen

Tie everything together with a tiny finite state machine. Three states are enough; the bonus game-over screen is just behavior of one of them.

StateWhat runsDisplay
IDLEinitialise snake (length, center), score = 0, seed/freeze foodstart screen or blank
PLAYtick → move, eat, grow, score, collision checkslive board + score
GAME OVERfreeze the snake; wait for restartgame-over screen (bonus)

The game-over screen (bonus, requirement 4.b)

You don't need fonts. Anything unmistakable works and earns the bonus:

  • Easiest: flood the whole visible area red (pixel_color = 12'hF00) while game_over is high.
  • Nicer: draw a big "X", a border frame, or a checkerboard, using simple XCoord/YCoord comparisons — the same rectangle math you used in Lab 6.
  • Showy: a hard-coded bitmap of the word "OVER" stored in a small ROM, indexed by the downsampled coordinate.
Restart cleanly. On returning to IDLE, re-initialise every piece of state — snake body, length, direction, score, and food — or your second game starts with the corpse of the first. An asynchronous reset (the Lab-5 metastability topic) that returns to IDLE is the simplest way.

12. Common Pitfalls (Lab 7 Hints!)

  1. Mixing up the three rates. The snake moves on the tick (~6 Hz), the screen is painted on the pixel clock (50 MHz), and logic is clocked at 100 MHz. Update game state only inside if (tick); never on every pixel.
  2. Turning more than once per tick. Buffer the keyboard request into dir_next and commit it once per tick, or fast presses let the snake reverse via two 90° turns. (requirement 3.b)
  3. Inverted Y axis. Up = row − 1, Down = row + 1, because VGA row 0 is at the top.
  4. Food spawning on the snake. Always re-roll the LFSR until the candidate cell is empty before latching the new food.
  5. Forgetting to grow correctly. Growth means not removing the tail on the eating tick (Approach A: length++; Approach B: bump the head value). If you move and grow you'll get an off-by-one body.
  6. Self-collision against the vacating tail. Decide whether the head may enter the cell the tail is leaving this tick, and check collision in the matching order.
  7. Leaking color into the blanking region. The Lab-6 rule still holds: force black unless hcount<800 && vcount<600.
  8. Hard-coded magic numbers. Rule #3 demands parametric modules — make CELL, GRID_W, GRID_H, MAX_LEN, TICK_MAX parameters so the design is reusable and easy to simulate.
  9. Not resetting all state on restart. Re-initialise snake, length, direction, score, and food when returning to IDLE.
  10. Missing keyboard one-shot. Reuse the Lab-5 fix: the keyboard resends make-codes while a key is held, so derive a single keyPressed pulse — don't act on every repeat.
If you internalize one sentence from this whole document: Snake is not a new protocol — it is a small grid-based state machine clocked by a slow "tick," wedged between the keyboard reader you built in Lab 5 and the screen painter you built in Lab 6; design the grid data structure first and everything else falls into place.

13.4 — Constraint files (.xdc) — merge two into one

Snake needs the pins of both previous labs at once: keyboard + 7-segment and VGA. So build the lab7 XDC by merging:

FilePins it defines

When merging XDC files: keep the clk and reset constraints only once (duplicates cause errors), and make sure every set_property port name matches a port on your Snake_Top exactly. Delete pin lines for any peripheral you don't actually use, and rename the file to whatever your top module expects.
Suggested build order using these files: start from LAB_06/VGA_Top.v (you get pixels on screen immediately), drop in Ps2_Interface from Lab 5 to read the keys, add the tick divider from the LAB_04 counters, then write the one genuinely new module — Snake_Logic — and finally hang Seg_7_Display off the score.
